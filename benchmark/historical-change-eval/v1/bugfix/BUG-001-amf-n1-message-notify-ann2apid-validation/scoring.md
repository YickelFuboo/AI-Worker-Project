# 评分规则

总分 100 分。

1. 需求理解，15 分：是否理解原始问题、业务意图或协议约束。
2. 仓库定位，10 分：是否只修改正确仓库，是否避免牵连无关仓库。
3. 文件/函数定位，15 分：是否命中真实人工修改所在的关键文件和函数。
4. 行为等价性，30 分：生成结果是否与 ground truth 行为等价；允许实现不同，但必须覆盖同样关键场景。
5. 最小性与项目风格，10 分：是否复用现有工具和错误风格，是否避免无关重构。
6. 测试/验证，15 分：是否补充或运行合适测试，验证是否可信。
7. 回归风险，5 分：是否避免引入新的 panic、错误响应、协议不兼容或状态污染。

评估时不要只做 patch 文本相似度比较。功能行为等价优先于 diff 完全一致；但如果实现绕开了项目既有模式、改动过大或遗漏关键边界，应相应扣分。
