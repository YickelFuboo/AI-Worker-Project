# 历史 MR 规则候选逆向提取

## 功能描述

逆向分析**一批历史 MR**（按时间段或 MR 列表选定）的代码 diff 与 PR review comments（检视意见），提炼出可指导后续正向开发的**编码规则与经验候选条目**，产出 `repos/{repo}/.agent/mr_rules_candidates/{YYYY-MM-DD}-{批次标识}.md`。该报告作为 `rev-repo-to-rules` 的**源 2 输入**，与源 1（代码仓扫描）合并去重后写入 `rules/` 目录（`约束/` / `模式/`）。

本 skill 是逆向 skill，分析的是**历史沉淀**，不是当前变更。区别于 `fwd-mr-impact-analysis`（正向链路单 MR 实时影响分析，喂 fwd-doc-sync / qa-artifact-auto-verify / fwd-regression-test-select）。

## 所属 Agent

实现逆向 Agent

## 适用场景

- 存量项目逆向还原时，需从历史 MR 的检视意见中提炼"老手才知道的坑与最佳实践"，避免后续 AI 生成的代码偏离团队既有要求
- 识别检视意见中反复强调但代码尚未统一的约定（沉淀为新规则）
- 识别历史 MR 中新沉淀的最佳实践模式（代码已落地但 `rules/` 目录未登记）
- 作为 `rev-repo-to-rules` 的源 2 输入，与代码扫描结果合并产 `rules/` 目录下文件

## 工作方式

### 执行步骤

1. **输入确认**：确认分析范围——时间段（如 2026-01-01 至 2026-06-01）或 MR ID 列表
2. **MR 清单收集**：按范围从代码托管平台（GitLab/GitHub）拉取 MR 列表，含 MR ID、标题、合入时间、关联需求/issue
3. **逐 MR 分析**（对范围内每个 MR）：
   - 拉取 MR 的代码 diff 与 commit message
   - 拉取 MR 的 PR review comments（检视意见）
   - **从检视意见提炼规则候选**：识别反复强调的约定（如"必须走封装层""必须配套取消""错误码必须上报"），区分"已落地"与"要求但代码尚未统一"
   - **从 diff 提炼新沉淀的最佳实践**：识别代码中新引入的成熟模式（如新的并发控制写法、新的错误处理封装），若该模式在 `rules/` 目录未登记，作为候选
4. **候选条目归并**：跨 MR 合并相似候选条目，去重；每条候选标注：
   - 拟归入 `rules/` 子目录与文件名（`约束/{主题}.md` / `模式/{动作或模式名}.md`），文件名按 `rev-repo-to-rules` 的命名约定拟一个候选名（中文，允许夹必要英文术语）
   - 来源 MR ID 列表（一条候选可能来自多个 MR 的检视意见）
   - 状态：已落地 / 检视要求未落地
   - 代码证据：`文件路径::符号名` 格式（已落地的给代码位置，未落地的给检视意见引用）
5. **置信度评估**：检视意见中明确要求且有代码证据的为高；仅检视意见提及无代码证据的为中；从 diff 模式推断的为中低
6. **生成候选报告**：按模板输出

### 注意事项

- 本 skill 只产**规则候选报告**，不直接写 `rules/` 目录；`rules/` 下文件的合并去重与最终写入由 `rev-repo-to-rules` 负责
- 本 skill 不做单 MR 影响面分析（那是 `fwd-mr-impact-analysis` 的正向职责）；本 skill 聚焦"从历史 MR 提炼规则候选"
- 候选条目需给出具体的代码证据或检视意见引用，避免泛化（如"注意并发安全"无意义，应写"AmfUe 上下文读写必须持 RWMutex，见 `amf_ue.go::AmfUe.SyncLock`"）
- 检视意见提炼时，过滤无意义评论（如"good""LGTM"），只保留有规则含义的内容
- 同一规则在多个 MR 检视意见中反复出现，合并为一条候选并标注"高频"，提升 rev-repo-to-rules 合并时的优先级
- 候选条目若与源 1（代码扫描）既有规则冲突，标注"待人工裁决"，由 rev-repo-to-rules 在合并阶段处理
- 增量场景：本批次产出的候选报告与历史候选报告独立存放，rev-repo-to-rules 消费时按时间戳取最新批次 + 历史未消费批次

### 输出要求

- 输出路径：`repos/{repo}/.agent/mr_rules_candidates/{YYYY-MM-DD}-{批次标识}.md`（一次性报告，不作为活文档维护；批次标识用时间段简写或 MR 范围哈希）
- 严格按模板格式输出：`templates/mr_rules_candidate_template.md`
- 报告包含：批次概述 / 分析范围 / 候选条目清单（按拟归入 `rules/` 子目录分组：`约束/` / `模式/`）/ 高频候选（多 MR 反复强调）/ 待人工裁决项 / 置信度
- 每条候选标注：拟归入子目录与文件名、来源 MR、状态、代码证据、置信度
- 高频候选单独列出，供 rev-repo-to-rules 优先合并
