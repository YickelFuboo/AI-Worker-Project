# 跨仓影响传播

## 功能描述

分析单个代码仓变更对其他代码仓的运行时和接口影响传播。超越 `multi-repo-dependency` 的编译依赖层面，深入到接口调用、消息流、共享数据等运行时影响，辅助评估跨仓回归和协同发布需求。

## 所属 Agent

架构设计 Agent（与 `arch-impact-analysis` 协同）

## 适用场景

- 多仓服务中某一仓的接口或消息格式变更
- 评估跨仓协同发布的需求
- 跨仓故障追溯（一仓变更引发他仓故障）
- 评估回滚某仓变更对其他仓的影响

## 工作方式

### 执行步骤

1. **读取变更仓的架构变更说明**：加载 `architecture_changes/` 下的接口契约变更
2. **读取元素依赖图**：加载 `architectures/logic_view/elements_tree.yaml`，构建跨仓元素依赖 DAG
3. **读取接口调用关系**：加载各元素的 `interfaces.yaml` 和 `dependencies.yaml`，识别接口的提供方和调用方
4. **读取跨视角缓存**：加载 `.agent/cross_perspective_cache.yaml` 中的场景→接口→代码映射
5. **影响传播分析**：
   - 直接影响：调用变更接口的其他仓元素
   - 间接影响：通过消息流、共享数据传播的下游仓
   - 循环依赖检测：识别潜在的双向影响回路
6. **生成影响传播报告**：输出受影响仓清单、影响路径、建议协同验证范围

### 注意事项

- 影响传播必须区分编译期（由 `multi-repo-dependency` 处理）和运行期（本 skill 处理）
- 破坏性接口变更的影响传播必须穷尽所有调用方
- 跨仓影响分析结果应作为 `regression-test-select` 的输入，扩展回归范围
- 跨仓协同发布需求应标注发布顺序约束

### 输出要求

- 输出影响传播图（PlantUML，标注影响路径和方向）
- 输出受影响仓清单：仓名、受影响元素、影响类型（接口/消息/数据）
- 输出协同发布建议：发布顺序、协同验证范围、回滚策略
