# 代码到业务场景挖掘

## 功能描述

从存量代码和设计文档中挖掘业务场景，反向生成 `features/` 目录下的业务视角内容。识别代码中实现的业务能力，提取子特性规格、架构关联和场景交互流程。

## 所属 Agent

业务逆向 Agent

## 适用场景

- 存量项目缺少业务文档，需要从代码反向理解业务
- 代码中有完整的实现但缺少对应的特性说明和场景描述
- 需要建立业务视角与代码实现的映射关系
- 存量反向分析流程的第一步

## 工作方式

### 执行步骤

**模式判定**：检查目标产物（`features/` 下子特性 spec.md / SCENARIO_*.md / arch_ref.yaml / features_tree.yaml）是否存在且 YAML 头部有 `last_modified`。存在 → 进入增量更新模式；不存在 → 进入首次生成模式。

#### 首次生成模式

1. **读取代码仓文档**：加载各代码仓的 `spec.md` 和 `design.md`，理解仓库定位和技术栈
2. **读取架构视图**：加载 `architectures/logic_view/elements_tree.yaml` 和相关元素规格，理解系统架构
3. **读取领域知识**：加载 `knowledge/领域知识/` 中的协议规范和常见模式，辅助识别代码中的业务概念
4. **扫描源码**：分析关键模块的入口函数、消息处理函数、状态机实现，识别业务场景
5. **场景提取**：从代码逻辑中提取用户场景（正常流程、异常流程、边界条件）
6. **生成特性文档**：
   - 生成/更新 `features/` 下的子特性 `spec.md`（业务说明、功能范围、关键规格）
   - 生成 `arch_ref.yaml`（关联架构元素和接口）
   - 生成场景交互流程文件（`SCENARIO_xxx_*.md`）
7. **更新索引**：更新 `features_tree.yaml` 和二级特性索引文件

#### 增量更新模式

1. **解析时间锚**：读取各产物 YAML 头部 `last_modified`，作为该产物的上次刷新时间点
2. **变更探测**：在各代码仓内执行 `git log --since="<last_modified>" --name-status --pretty=format:` 与 `git diff <commit_at_last_modified>..HEAD --name-status`，得到新增/修改/删除文件清单（按仓分组）
3. **变更分类**：按文件路径归入影响域：
   - 入口函数/消息处理函数/状态机文件 → 可能影响场景交互流程
   - 模块对外接口签名变更 → 可能影响 `arch_ref.yaml`
   - 新增业务模块或大改定位 → 可能影响子特性 spec.md 业务功能范围
   - 纯配置/工具/日志改动 → 通常不影响业务场景
4. **刷新决策**：仅对受影响域执行增量分析；无变更影响域跳过。判定结论：
   - 无变更 → 不刷新，更新产物 `last_modified` 为本次检查时间并记录"无变更"到逆向报告
   - 局部变更 → 仅刷新受影响章节（如 SCENARIO 文件、arch_ref 中的接口映射）
   - 结构性变更（如新增子特性、状态机重构）→ 触发对应子特性的首次生成式全量刷新
5. **刷新执行**：对需刷新的章节，重新读取相关源码文件（仅变更文件 + 其直接依赖），与现有 md 内容比对，合并更新；保留未受影响章节原内容
6. **更新索引与时间戳**：更新 `features_tree.yaml`、二级索引、各产物 `last_modified` 为本次刷新时间
7. **生成增量报告**：列出本次刷新的产物清单、变更文件清单、跳过的章节与原因、置信度变化

### 注意事项

- 业务场景挖掘需结合领域知识，将代码概念映射到业务术语
- 生成的场景必须标注置信度（高/中/低），低置信度需人工审核
- 场景交互流程文件中的步骤应与代码调用链一致
- 参考 `knowledge/模板库/` 中的 spec 和场景模板格式
- 增量模式下，`git log --since` 依赖提交时间，若仓库有 rebase/squash 导致时间漂移，需用 `git log --before="<last_modified>" -1 --format=%H` 先定位锚点提交，再用 `git diff <锚点提交>..HEAD`
- 删除文件若对应已废弃场景，需同步从 `features_tree.yaml` 与索引中移除，不得遗留

### 输出要求

- 输出 `features/` 下新增或更新的子特性目录：
  - 子特性 `spec.md`：按工作空间 2.3.3 节定义的子特性规格文件格式（YAML front matter + 业务概述/功能范围/关键规格/业务场景/架构依赖）
  - 场景交互流程 `SCENARIO_*.md`：严格按模板 `templates/scenario_flow_template.md`
  - `arch_ref.yaml`：按工作空间 2.3.3 节定义的格式
- 更新 `features_tree.yaml` 和二级索引文件
- 输出场景提取报告：提取的场景列表、置信度评估、代码映射
