# 代码到业务场景挖掘

## 功能描述

从存量代码和设计文档中挖掘业务场景，反向生成 `features/` 目录下的业务视角内容。识别代码中实现的业务能力，提取子特性规格、架构关联和场景交互流程。

## 所属 Agent

业务逆向 Agent

## 适用场景

- 存量项目缺少业务文档，需要从代码反向理解业务
- 代码中有完整的实现但缺少对应的特性说明和场景描述
- 需要建立业务视角与代码实现的映射关系
- 存量反向分析流程的第一步

## 工作方式

### 执行步骤

1. **读取代码仓文档**：加载各代码仓的 `spec.md` 和 `design.md`，理解仓库定位和技术栈
2. **读取架构视图**：加载 `architectures/logic_view/elements_tree.yaml` 和相关元素规格，理解系统架构
3. **读取领域知识**：加载 `knowledge/领域知识/` 中的协议规范和常见模式，辅助识别代码中的业务概念
4. **扫描源码**：分析关键模块的入口函数、消息处理函数、状态机实现，识别业务场景
5. **场景提取**：从代码逻辑中提取用户场景（正常流程、异常流程、边界条件）
6. **生成特性文档**：
   - 生成/更新 `features/` 下的子特性 `spec.md`（业务说明、功能范围、关键规格）
   - 生成 `arch_ref.yaml`（关联架构元素和接口）
   - 生成场景交互流程文件（`SCENARIO_xxx_*.md`）
7. **更新索引**：更新 `features_tree.yaml` 和二级特性索引文件

### 注意事项

- 业务场景挖掘需结合领域知识，将代码概念映射到业务术语
- 生成的场景必须标注置信度（高/中/低），低置信度需人工审核
- 场景交互流程文件中的步骤应与代码调用链一致
- 参考 `knowledge/模板库/` 中的 spec 和场景模板格式

### 输出要求

- 输出 `features/` 下新增或更新的子特性目录：
  - 子特性 `spec.md`：按工作空间 2.3.3 节定义的子特性规格文件格式（YAML front matter + 业务概述/功能范围/关键规格/业务场景/架构依赖）
  - 场景交互流程 `SCENARIO_*.md`：严格按模板 `templates/scenario_flow_template.md`
  - `arch_ref.yaml`：按工作空间 2.3.3 节定义的格式
- 更新 `features_tree.yaml` 和二级索引文件
- 输出场景提取报告：提取的场景列表、置信度评估、代码映射
