# 变更日志管理

## 功能描述

为每次代码变更生成结构化的变更日志文件。记录变更时间、变更内容、影响范围、测试情况和文档同步状态，形成可追溯的变更历史。

## 适用场景

- 完成一次代码修改后，需要记录变更历史
- 发布新版本前，需要汇总变更列表
- 需要追溯某个功能是什么时候、由谁、为什么修改的
- 对应参考规则中的"完备的变更日志管理"

## 工作方式

### 执行步骤

1. **收集变更信息**：
   - 从 `git diff` 获取变更的文件列表
   - 识别变更类型：新增功能 / Bug 修复 / 重构 / 文档更新 / 配置变更
   - 获取关联的需求/issue 编号

2. **编写变更日志**：
   - 按模板格式输出
   - 简要说明改动内容、影响范围
   - 记录测试执行情况和结果
   - 记录文档同步情况

3. **保存日志文件**：
   - 文件命名：`YYYY-MM-DD-{主题}.md`
   - 保存到 `architectures/changeLogs/` 目录

### 注意事项

- 变更日志聚焦"改了什么、为什么改、影响谁"，不重复代码细节
- 如果本次变更有关联的 Spec/Design 文档更新，在日志中引用
- 如果测试未全部通过，在日志中明确说明原因和残留风险
- 一次提交聚合一次变更，一个变更日志文件对应一次完整的改动

### 输出要求

- 日志文件：`architectures/changeLogs/YYYY-MM-DD-{主题}.md`
- 内容包含：变更概述、文件变更清单、影响范围、测试情况、文档同步
