# 正向全流程开发编排

## 功能描述

正向流程的顶层编排入口：接收用户的自然语言需求，依次执行需求分析 → 规格说明书 → 设计文档 → 改码点定位 → 代码实现 → 测试验证 → 变更日志。适用于从零开始的新功能开发。

## 适用场景

- 用户提出新功能需求，需要端到端完成开发
- 需要保证每个阶段输出物完整、可追溯
- 需要严格遵循正向流程规范
- 正向流程的总入口

## 工作方式

### 执行步骤

#### 阶段一：需求分析

1. 执行 `req-analysis`：将用户诉求转化为结构化需求文档
2. 输出：`requirements/{feature_name}_requirements.md`
3. 检查点：需求边界是否明确、验收标准是否可测试

#### 阶段二：规格说明书

1. 执行 `write-spec`：将需求转化为功能规格说明书
2. 输出：`requirements/specs/{feature_name}_spec.md`
3. 检查点：接口契约是否精确、数据字段是否有校验规则

#### 阶段三：设计文档

1. 执行 `write-design`：将规格转化为技术设计
2. 输出：`architectures/modules/{module_name}/design.md`（模块级）或更新仓级设计
3. 检查点：模块划分是否合理、接口定义是否与 Spec 一致

#### 阶段四：改码点定位

1. 执行 `locate-change-point`：L1→L2→L3 定位改码点
2. 输出：明确的 `(file_path, class_name | function_name)`
3. 检查点：定位理由是否清晰、影响面是否已评估

#### 阶段五：代码实现

1. 执行 `spec-driven-dev`：三阶段（接口规格→测试→代码）
2. 或执行 `generate-code`：基于模式生成代码
3. 输出：源代码 + 测试代码
4. 检查点：测试全部通过

#### 阶段六：质量保障

1. 执行 `code-smell-gate`：MR 前坏味道检查
2. 输出：门禁结论
3. 检查点：无未修复的"必须修复"项

#### 阶段七：文档同步与变更记录

1. 执行 `doc-sync`：同步更新受影响的文档
2. 执行 `change-log`：生成变更日志
3. 输出：变更日志文件 + 更新的文档

### 注意事项

- 每个阶段完成后，确认输出物完整再进入下一阶段
- 如果某阶段发现上游问题，回退修正，不可跳过
- 如果项目尚无图谱（存量项目），应先执行 `reverse-generate-all` 建立基准
- 如果图谱已过期（代码变更后未增量更新），先执行 `graph-incremental-update`
- 每阶段输出物路径应保持约定，便于后续追溯

### 停止原则

以下情况应暂停流程：
- 需求分析阶段发现关键不确定性且用户无法确认
- 设计阶段发现需要破坏性变更但未评估影响范围
- 实现阶段发现设计缺陷（回退到设计阶段）
- 测试失败且非显而易见原因（不可为通过而弱化测试）

## 输出要求

流程完成后汇总：

| 阶段 | 输出物 | 路径 | 状态 |
|------|--------|------|------|
| 需求分析 | 需求文档 | `requirements/{name}_requirements.md` | ✅/❌ |
| 规格说明书 | Spec | `requirements/specs/{name}_spec.md` | ✅/❌ |
| 设计文档 | Design | `architectures/...` | ✅/❌ |
| 改码定位 | 定位结果 | — | ✅/❌ |
| 代码实现 | 源码+测试 | 按项目约定路径 | ✅/❌ |
| 质量门禁 | 检查报告 | — | PASS/FAIL |
| 变更日志 | 日志文件 | `architectures/changeLogs/...` | ✅/❌ |
