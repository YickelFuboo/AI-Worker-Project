# MR 前代码坏味道检查

## 功能描述

在提交 MR（Merge Request）前检查代码坏味道。使用 `git diff` 获取本次修改范围，识别代码坏味道，按严重程度分级（必须修复、建议修复、记录记忆），先修复必须修复项，修改后做最小复检，最后输出 MR 门禁结论。

## 适用场景

- 开发完成准备提交 MR
- 代码 Review 前的自检
- CI 流水线中的质量门禁
- 对应参考规则中的 `pre-mr-code-smell-gate`

## 工作方式

### 执行步骤

1. **获取变更范围**：
   - 执行 `git diff` 获取本次修改的文件和变更内容
   - 识别新增文件、修改文件、删除文件

2. **识别代码坏味道**：
   按检查项清单逐项检查：
   - 过长函数（> 建议行数）
   - 过长参数列表（> 建议参数数）
   - 重复代码
   - 过深嵌套
   - 魔法数字
   - 未处理的错误返回
   - 过大文件
   - 注释掉的代码
   - 不一致的命名风格
   - 硬编码的配置值

3. **分级**：
   - **必须修复**：安全漏洞、逻辑错误、破坏性 API 变更未标注
   - **建议修复**：可读性差、可维护性问题、性能隐患
   - **记录记忆**：风格偏好、可改进但不阻塞 MR 的项

4. **修复必须修复项**：
   - 逐个修复必须修复项
   - 每次修复后运行相关测试

5. **最小复检**：
   - 确认必须修复项已全部修复
   - 确认修复未引入新问题

6. **输出门禁结论**：
   - PASS：无必须修复项
   - PASS WITH NOTES：有建议修复项或记录项
   - FAIL：存在未修复的必须修复项

### 注意事项

- 只检查本次 diff 涉及的代码，不检查存量代码
- 修复后必须运行测试，确认不影响已有功能
- 分级参考 `references/smell_checklist.md`

### 输出要求

```
MR 代码坏味道检查报告

变更范围: {文件数} 文件, +{新增行} -{删除行}

必须修复 (X):
  - [ ] {文件:行号} {问题描述} → {修复建议}
  修复状态: 已修复 / 未修复

建议修复 (Y):
  - [ ] {文件:行号} {问题描述} → {修复建议}

记录记忆 (Z):
  - {文件:行号} {问题描述}

门禁结论: PASS / PASS WITH NOTES / FAIL
```
