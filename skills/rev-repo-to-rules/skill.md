# 仓级特有规则提取

## 功能描述

从存量代码仓中逆向提取本仓特有的约束规则（第三方库版本限制、接口使用约束、已知技术债、性能注意事项等），生成 `repos/{仓名}/.agent/rules.md`。该文件为可选产物，仅当仓存在与全局 `knowledge/编码规范/` 公共规范不同的细化约束时才生成，且不可与全局规范冲突。

## 所属 Agent

实现逆向 Agent

## 适用场景

- 存量项目逆向还原时，发现仓内存在全局编码规范未覆盖的特有约束
- 需要将仓内的隐性约定（如某接口必须走封装层、某第三方库版本下限）显式化为可被质量质检 Agent 校验的规则
- 需要登记已知技术债，便于后续版本规划
- 作为 `rev-code-to-spec-and-design` 的补充，在仓级 `spec.md` / `design.md` 生成后按需触发

## 工作方式

### 执行步骤

**模式判定**：检查 `repos/{仓名}/.agent/rules.md` 是否存在且 YAML 头部有 `last_modified`。存在 → 进入增量更新模式；不存在 → 进入首次生成模式（含"扫描后无特有约束则不生成"判定）。

#### 首次生成模式

1. **前置检查**：确认 `repos/{仓名}/.agent/spec.md` 已生成，理解仓的定位、技术栈与外部依赖
2. **加载全局规范**：读取 `knowledge/编码规范/` 下的公共规范（语言风格、安全规则、依赖管理策略等），作为基准
3. **扫描源码与配置**：
   - `go.mod` / `package.json` / `requirements.txt` / `pom.xml` 等依赖声明文件 → 识别第三方库版本下限与锁定要求
   - 模块入口与对外接口函数 → 识别"必须走封装层"类接口使用约束
   - `TODO` / `FIXME` / `XXX` 注释与已知 issue 标记 → 识别已知技术债
   - 性能敏感路径（如热循环、锁竞争点）注释或 benchmark 标记 → 识别性能注意事项
4. **差异比对**：将扫描结果与全局规范比对，仅保留全局规范未覆盖或需细化的条目；冲突项不得写入（应由人工决策全局规范是否修订）
5. **规则分类**：按"第三方库限制 / 接口使用约束 / 已知技术债 / 性能注意事项"四类组织
6. **生成产物**：按模板输出 `repos/{仓名}/.agent/rules.md`；若扫描后无任何特有约束，则不生成文件并记录到逆向报告中
7. **更新报告**：在本次执行的差异摘要中补充 rules.md 的提取结论与置信度（一次性产物，不写入 rules.md 正文）

#### 增量更新模式

1. **解析时间锚**：读取 `rules.md` YAML 头部 `last_modified`
2. **定位锚点提交**：`git log --before="<last_modified>" -1 --format=%H`
3. **变更探测**：`git diff <锚点提交>..HEAD --name-status`，重点关注：
   - 依赖声明文件变更 → 可能新增第三方库限制条目
   - 接口/封装层文件变更（新增、签名修改、删除）→ 可能影响接口使用约束
   - 新增/移除 `TODO`/`FIXME`/`XXX` 注释 → 已知技术债清单变化
   - 新增 benchmark 文件或性能注释 → 性能注意事项变化
4. **变更分类与章节映射**：
   - 依赖声明新增库或版本提升 → 第三方库限制（追加或更新版本约束）
   - 新增封装层 → 接口使用约束（追加"必须走 X 封装层"条目）
   - 既有封装层删除 → 接口使用约束（移除对应条目，确认调用方已迁移）
   - 新增 TODO/FIXME → 已知技术债（追加，标注发现时间）
   - 既有 TODO/FIXME 被移除（代码已修复）→ 已知技术债（移除条目，记录"已修复"）
   - 性能注释/benchmark 变更 → 性能注意事项
5. **刷新决策**：
   - 无变更 → 不刷新，仅更新 `last_modified` 为检查时间并记录"无变更"
   - 局部变更 → 仅刷新受影响章节，保留未变条目
   - 全局规范更新导致差异点变化 → 重新执行差异比对，可能新增或移除多条
6. **刷新执行**：按模板比对现有内容，追加/更新/移除条目；与全局规范冲突的条目仍不得写入
7. **更新时间戳与报告**：更新 `last_modified`，输出差异摘要（新增/更新/移除条目清单、与全局规范的差异点变化、置信度变化）

### 注意事项

- rules.md 仅记录**本仓特有**约束，不得重复全局规范已有内容
- 与全局规范冲突时不得写入 rules.md，应在逆向报告中标注冲突并提请人工裁决
- 已知技术债需标注影响范围与拟修复版本（若代码中有线索），无线索时标注"待评估"
- 接口使用约束需给出具体的封装层路径或函数签名，避免泛化描述
- 第三方库限制需给出具体版本号与约束原因（如"≥ 1.5.0，低版本不支持扩展 AVP"）
- 产物置信度按"高/中/低"标注，源码中有明确版本声明或注释的为高，依据调用模式推断的为中低
- 增量模式下，已修复的技术债（TODO 注释消失）必须从清单移除并记录"已修复"，不得遗留
- 封装层删除若未确认调用方迁移，应保留条目并标注"待人工确认调用方迁移完整性"，避免误删

### 输出要求

- 输出路径：`repos/{仓名}/.agent/rules.md`（无特有约束时不生成）
- 严格按模板格式输出：`templates/repo_rules_template.md`
- YAML 元数据头部包含 `repo_id`、`last_modified`、`last_modified_by`（写入执行 skill 名，如 `rev-repo-to-rules` / `fwd-doc-sync`）、`confidence`
- 正文按"第三方库限制 / 接口使用约束 / 已知技术债 / 性能注意事项"分节，无内容的节写"无"
- 同步在逆向报告中记录：是否生成 rules.md、提取的条目数、置信度评估、与全局规范的差异点
