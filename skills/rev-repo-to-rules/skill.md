# 仓级编码规则与经验提取

## 功能描述

从**两路输入**逆向提炼本仓的**编码实现规则与经验**，生成 `repos/{仝名}/.agent/rules.md`：
- **源 1：代码仓扫描**——提炼已落地的编码约定（"代码里就是这么写的"）
- **源 2：历史 MR 规则候选报告**——消费 `rev-repo-mr-to-rules-candidate` 产出的 `repos/{仝名}/.agent/mr_rules_candidates/` 下报告，提炼其中已归并的规则候选条目（"检视意见里要求但代码还没统一的"或"新沉淀的最佳实践"）

两路输入合并去重后统一写入 rules.md。该文件聚焦"如何在这个仓里写好代码"，包含第三方库使用规则、接口使用约定、编码风格特有约定、实现模式与经验、性能注意事项、可观测性约定、文档同步约定、已知技术债，供 `fwd-generate-code` / `fwd-code-compliance-check` / `fwd-ut-generate` 等正向 Skill 作为编码约束与经验参考。始终生成（即使各节均"无"，也保留文件声明"本仓遵循全局编码规范无例外"）。

本 skill 是 rules.md 的**唯一写入者**；历史 MR 规则候选提炼由 `rev-repo-mr-to-rules-candidate` 负责，本 skill 只消费其产物，不直接分析 MR diff 与检视意见。

## 所属 Agent

实现逆向 Agent

## 适用场景

- 存量项目逆向还原时，需提炼仓内隐性编码规则与经验，避免后续 AI 生成的代码偏离既有风格
- 需要将"老手才知道的坑与最佳实践"显式化，如某接口必须走封装层、某定时器必须配套取消、某错误码必须上报
- 作为 `rev-repo-to-spec-and-design` 的补充，在仓级 `spec.md` / `design.md` 生成后触发
- 正向开发阶段，`fwd-generate-code` 在写代码前加载本文件作为编码约束

## 工作方式

### 执行步骤

**模式判定**：检查 `repos/{仓名}/.agent/rules.md` 是否存在且 YAML 头部有 `last_modified`。存在 → 进入增量更新模式；不存在 → 进入首次生成模式。

#### 首次生成模式

1. **前置检查**：确认 `repos/{仓名}/.agent/spec.md` 和 `design.md` 已生成，理解仓的业务功能、模块划分、接口契约
2. **加载全局编码规范**：读取 `knowledge/编码规范/` 下的公共规范，作为基准；本文件仅记录全局规范未覆盖或需细化的条目
3. **源 1 扫描代码仓提炼规则与经验**：
   - `go.mod` / `package.json` 等依赖声明 → 第三方库使用规则（版本约束、使用边界、禁用项）
   - 模块入口与对外接口 → 接口使用约定（必须走封装层、禁止直接调底层、入参校验位置）
   - error 处理、日志、context 传递、并发控制、资源释放 → 编码风格特有约定
   - 既有代码中的成熟模式（如定时器配套取消、状态机转换守卫、SBI 重试退避）→ 实现模式与经验
   - 性能敏感路径注释、benchmark、锁竞争点 → 性能注意事项
   - `TODO` / `FIXME` / `XXX` 注释 → 已知技术债
4. **源 2 扫描历史 MR 规则候选报告**：
   - 扫描 `repos/{仝名}/.agent/mr_rules_candidates/` 下所有候选报告，提取已按 rules.md 章节归类的候选条目（该报告由 `rev-repo-mr-to-rules-candidate` 逆向分析历史 MR 代码 diff 与 PR review comments 提炼而成）
   - 优先处理"高频候选"（多 MR 反复强调的规则）
   - 处理"待人工裁决项"中与源 1 既有规则冲突的条目
5. **两路合并去重**：源 1 与源 2 提炼的规则按"规则内容相似度"去重；同一规则两源都有时合并代码证据；源 2 独有的标注"来源：MR 检视"
6. **差异比对**：与全局编码规范比对，仅保留细化或新增条目；冲突项不写入（提请人工裁决全局规范是否修订）
7. **生成产物**：按模板输出 `repos/{仝名}/.agent/rules.md`，各节无内容写"无"
8. **输出差异摘要**：一次性产物，记录两路输入的条目数、去重情况、置信度、与全局规范的差异点

#### 增量更新模式

1. **解析时间锚**：读取 `rules.md` YAML 头部 `last_modified`
2. **源 1 代码变更探测**：`git log --before="<last_modified>" -1 --format=%H` 定位锚点提交 → `git diff <锚点提交>..HEAD --name-status`
3. **源 2 候选报告新增探测**：扫描 `mr_rules_candidates/` 下时间晚于 `last_modified` 的新增候选报告，提取候选条目
4. **变更分类与章节映射**（两源合并）：
   - 依赖声明变更（源 1） → 第三方库使用规则
   - 封装层/接口签名变更（源 1） → 接口使用约定
   - error/log/context/concurrency 模式变化（源 1） → 编码风格特有约定
   - 新增成熟模式或既有模式被修改（源 1） → 实现模式与经验
   - 性能注释/benchmark 变更（源 1） → 性能注意事项
   - TODO/FIXME 增减（源 1） → 已知技术债（消失则记录"已修复"）
   - MR 报告新增 rules.md 建议（源 2） → 对应章节追加或更新，标注"来源：MR 检视"
5. **去重合并**：源 1 与源 2 本周期新增条目去重，避免同一规则重复登记
6. **刷新决策**：两源均无变更→仅更新时间戳；任一源有变更→刷新对应章节；全局编码规范更新→重做差异比对
7. **更新时间戳与报告**

### 注意事项

- rules.md 聚焦"如何在这个仓里写好代码"，是编码规则与经验的载体，不是业务规格（归 spec.md）、不是设计说明（归 design.md）、不是测试基础设施（归 DTFrame.md）
- 与全局编码规范冲突时不得写入，提请人工裁决
- 实现模式与经验需给出具体代码示例或封装层路径，避免泛化（如"注意并发安全"无意义，应写"AmfUe 上下文读写必须持 RWMutex，见 `amf_ue.go::AmfUe.SyncLock`"）
- 已知技术债需标注影响范围与拟修复版本，无线索标注"待评估"
- 始终生成文件，即使各节均"无"，也保留文件以声明"本仓遵循全局编码规范无例外"
- 增量模式下，TODO 消失（代码已修复）必须从技术债移除并记录"已修复"
- **两路输入合并去重**：源 1（代码）与源 2（MR 报告/检视意见）提炼的规则按内容相似度去重；同一规则两源都有时合并代码证据；源 2 独有的条目标注"来源：MR 检视"，置信度默认中（待代码落地后升为高）
- 本 skill 只消费 `rev-repo-mr-to-rules-candidate` 产出的规则候选报告，不直接分析 MR diff 与检视意见；候选报告缺失时仅靠源 1
- 源 2 提炼的规则若与源 1 既有规则冲突，以源 1（已落地代码）为准，源 2 条目记入差异摘要待人工裁决
- 置信度：源码中有明确注释/封装层/版本声明的为高，依据调用模式推断的为中低，源 2 独有条目默认中

### 输出要求

- 输出路径：`repos/{仓名}/.agent/rules.md`（始终生成）
- 严格按模板格式输出：`templates/repo_rules_template.md`
- YAML 元数据头部包含 `repo_id`、`last_modified`、`last_modified_by`（写入执行 skill 名）、`confidence`
- 正文分节：第三方库使用规则 / 接口使用约定 / 编码风格特有约定 / 实现模式与经验 / 性能注意事项 / 可观测性约定 / 文档同步约定 / 已知技术债，无内容的节写"无"
- 每条规则给出具体的代码证据（`文件路径::符号名` 格式）或代码示例
- 同步在差异摘要中记录：提取条目数、置信度评估、与全局编码规范的差异点
