# 仓级编码规则与经验提取

## 功能描述

从**两路输入**逆向提炼本仓的**编码实现规则与经验**,生成 `repos/{repo}/.agent/rules/` 目录,按关注点拆分为单文件,供 AI 按文件名精准加载:
- **`约束/`** — 长期不变的编码约束(第三方库使用/接口调用/编码风格/性能/可观测性/文档同步)
- **`模式/`** — 修改模式与实现模式(如 `新增告警.md` / `新增SBI接口.md` / `定时器配套取消.md`,从 MR 归纳 + 代码扫描沉淀)

单文件 `rules.md` 的问题:AI 每次得读全文(浪费上下文),且多类内容混在一起语义层次不清。目录化后每个文件单一关注点,AI 按任务关键词匹配文件名加载,且每条规则独立演进、独立刷新。

**两路输入**:
- **源 1:代码仓扫描** — 提炼已落地的编码约定("代码里就是这么写的")
- **源 2:历史 MR 规则候选报告** — 消费 `rev-repo-mr-to-rules-candidate` 产出的 `repos/{repo}/.agent/mr_rules_candidates/` 下报告,提炼其中已归并的规则候选条目("检视意见里要求但代码还没统一的"或"新沉淀的最佳实践")

两路输入合并去重后按关注点拆分写入对应文件。本 skill 是 `rules/` 目录的**唯一写入者**;历史 MR 规则候选提炼由 `rev-repo-mr-to-rules-candidate` 负责,本 skill 只消费其产物,不直接分析 MR diff 与检视意见。

## 所属 Agent

实现逆向 Agent

## 适用场景

- 存量项目逆向还原时,需提炼仓内隐性编码规则与经验,避免后续 AI 生成的代码偏离既有风格
- AI 在代码生成 / 检视阶段按需加载特定规则文件(如新增告警时只读 `模式/新增告警.md`),避免读全文浪费上下文
- 需要将"老手才知道的坑与最佳实践"显式化,如某接口必须走封装层、某定时器必须配套取消、某错误码必须上报
- MR 检视意见沉淀的修改模式需要从候选报告转为正式规则
- 作为 `rev-repo-to-spec-and-design` 的补充,在仓级 `spec.md` / `design.md` 生成后触发
- 正向开发阶段,`fwd-generate-code` / `fwd-code-compliance-check` / `fwd-ut-generate` 按需加载 rules/ 下相关文件

## 工作方式

### 目录结构

```
repos/{repo}/.agent/rules/
├── _index.md             # 目录索引,列出所有规则文件 + 摘要 + 置信度
├── 约束/                  # 长期不变的编码约束(代码扫描沉淀,MR 检视补充)
│   ├── 第三方库.md          # 第三方库使用规则
│   ├── 接口使用.md          # 接口使用约定
│   ├── 编码风格.md          # 编码风格特有约定(error/log/context/concurrency/resource)
│   ├── 性能.md             # 性能注意事项
│   ├── 可观测性.md          # 可观测性约定(日志/指标/追踪/告警)
│   └── 文档同步.md          # 文档同步约定
└── 模式/                  # 修改模式与实现模式(按动作或模式名命名)
    ├── {动作或模式名}.md    # 如 新增告警.md / 新增SBI接口.md / 定时器配套取消.md
    └── ...
```

**子目录与文件命名约定**(子目录与文件名都用中文,文件名要能被 AI 按任务关键词检索):
- 子目录固定两个:`约束/` 与 `模式/`,不得新增或改名
- `约束/`:用名词或名词短语,如 `命名.md` / `错误处理.md` / `可观测性.md`;允许中文夹必要英文术语,如 `govalidator自定义tag注册.md`
- `模式/`:用动作短语或模式名,如 `新增告警.md` / `新增SBI接口.md` / `定时器配套取消.md` / `SBI重试退避.md`;允许中文夹必要英文术语
- 文件名禁止使用 OS 保留字符(`\/:*?"<>|`),禁止前后空白
- 文件名优先用动宾或主谓结构,提升 AI 按任务关键词匹配命中率

### 执行步骤

**模式判定**:检查 `repos/{repo}/.agent/rules/_index.md` 是否存在且 YAML 头部有 `last_modified`。存在 → 增量更新模式;不存在 → 首次生成模式。

#### 首次生成模式

1. **前置检查**:确认 `repos/{repo}/.agent/spec.md` 和 `design.md` 已生成,理解仓的业务功能、模块划分、接口契约
2. **加载全局编码规范**:读取 `knowledge/编码规范/` 下的公共规范,作为基准;本目录仅记录全局规范未覆盖或需细化的条目
3. **源 1 代码仓扫描,按关注点归类到对应文件**:
   - `go.mod` / `package.json` 等依赖声明 → `约束/第三方库.md`(版本约束、使用边界、禁用项)
   - 模块入口与对外接口 → `约束/接口使用.md`(必须走封装层、禁止直接调底层、入参校验位置)
   - error 处理、日志、context 传递、并发控制、资源释放 → `约束/编码风格.md`
   - 性能敏感路径注释、benchmark、锁竞争点 → `约束/性能.md`
   - 日志 / 指标 / 追踪 / 告警的实现约定 → `约束/可观测性.md`
   - 文档同步要求(从 spec / interfaces / design 交叉关系反推) → `约束/文档同步.md`
   - 既有代码中的成熟模式(定时器配套取消、状态机转换守卫、SBI 重试退避、批量处理) → `模式/{模式名}.md`,每个模式独立成文件
4. **源 2 MR 候选报告消费**:
   - 扫描 `repos/{repo}/.agent/mr_rules_candidates/` 下所有候选报告,提取候选条目(该报告由 `rev-repo-mr-to-rules-candidate` 逆向分析历史 MR 代码 diff 与 PR review comments 提炼而成,已按主题归类)
   - **修改模式类候选**(如"新增告警需改 A/B/C 文件") → 新建 `模式/{动作}.md` 或追加到既有模式文件
   - **编码约束类候选**(代码还没统一的检视意见) → 归入对应 `约束/{主题}.md`,frontmatter `sources` 含 `mr-candidate`,条目末尾标注"来源:MR 检视"
   - 优先处理"高频候选"(多 MR 反复强调的规则)
   - 处理"待人工裁决项"中与源 1 既有规则冲突的条目(不写入,记入差异摘要)
5. **两路合并去重**:
   - 源 1 与源 2 提炼的规则按"规则内容相似度"去重
   - 同一规则两源都有:合并代码证据,源 2 不另立文件,`sources` 含两路标识
   - 源 2 独有:新建文件,`sources: [mr-candidate]`,置信度默认中(待代码落地后升为高)
   - 源 1 独有:正常建立,`sources: [code-scan]`,置信度按代码证据强度
6. **差异比对**:与全局编码规范比对,仅保留细化或新增条目;冲突项不写入(提请人工裁决全局规范是否修订)
7. **生成产物**:
   - 按 templates 生成 `约束/` / `模式/` 下各文件
   - `约束/` 下 6 个基础文件始终生成(即使内容为"无"),作为仓级规则骨架
   - `模式/` 按实际内容生成,可为空
   - 生成 `_index.md` 索引,列出所有文件 + 标题 + 一句话摘要 + 置信度
8. **输出差异摘要**:记录两路输入的条目数、文件拆分情况、去重情况、置信度、与全局规范的差异点

#### 增量更新模式

1. **解析时间锚**:读取 `_index.md` YAML 头部 `last_modified`
2. **源 1 代码变更探测**:`git log --before="<last_modified>" -1 --format=%H` 定位锚点提交 → `git diff <锚点提交>..HEAD --name-status`
3. **源 2 候选报告新增探测**:扫描 `mr_rules_candidates/` 下时间晚于 `last_modified` 的新增候选报告
4. **按文件粒度变更映射**:
   - 依赖声明变更 → `约束/第三方库.md`
   - 封装层/接口签名变更 → `约束/接口使用.md`
   - error/log/context/concurrency 模式变化 → `约束/编码风格.md`
   - 性能注释/benchmark 变更 → `约束/性能.md`
   - 日志/指标/追踪/告警实现变化 → `约束/可观测性.md`
   - 文档同步要求变化 → `约束/文档同步.md`
   - 新增成熟模式或既有模式被修改 → 新建 `模式/{模式名}.md` 或刷新既有文件
   - MR 报告新增修改模式候选 → 新建 / 追加 `模式/{动作}.md`,标注"来源:MR 检视"
5. **去重合并**:源 1 与源 2 本周期新增条目去重,避免同一规则重复登记
6. **刷新决策**:
   - 两源均无变更 → 仅更新 `_index.md` 时间戳
   - 某文件相关变更 → 仅刷新该文件,其他文件不动
   - 全局编码规范更新 → 重做差异比对,可能影响多个 `约束/` 文件
   - 文件数变化 ≥30% 或重构 → 触发首次生成式全量刷新
7. **更新 `_index.md`**:文件增删必须同步刷新索引;`_index.md` 的 `last_modified` 每次执行都更新
8. **更新时间戳与报告**:输出差异摘要

### 注意事项

- **rules/ 是目录,不是单文件**:禁止把所有规则塞进一个 `rules.md`;按关注点拆分是为了 AI 精准加载,单文件模式已废弃
- **单一关注点**:每个文件只承载一个关注点(约束)或一个模式,避免大杂烩;若一个约束文件内容过载(如 `编码风格.md` 同时讲 error/log/concurrency/resource 四类且每类都很长),考虑拆分为 `编码风格-错误.md` / `编码风格-日志.md` 等
- **文件命名可被 AI 检索**:文件名要能体现内容,如 `新增告警.md` 比 `告警相关.md` 更利于 AI 按任务关键词匹配
- **子目录名固定**:只允许 `约束/` 与 `模式/` 两个子目录,且名称使用中文(不得使用 `constraints/` `patterns/` 英文别名)
- **`_index.md` 必须维护**:索引是 AI 决定加载哪些文件的入口,文件增删必须同步刷新;索引缺失或不准则 AI 检索失效
- **rules/ 聚焦"如何在这个仓里写好代码"**,是编码规则与经验的载体,不是业务规格(归 `spec.md`)、不是设计说明(归 `design.md`)、不是测试基础设施(归 `DTFrame.md`)
- **与全局编码规范冲突时不得写入**,提请人工裁决
- **实现模式需给出具体代码示例或封装层路径**,避免泛化(如"注意并发安全"无意义,应写"AmfUe 上下文读写必须持 RWMutex,见 `amf_ue.go::AmfUe.SyncLock`")
- **两路输入合并去重**:源 1(代码)与源 2(MR 报告/检视意见)提炼的规则按内容相似度去重;同一规则两源都有时合并代码证据;源 2 独有条目标注"来源:MR 检视",置信度默认中
- 本 skill 只消费 `rev-repo-mr-to-rules-candidate` 产出的规则候选报告,不直接分析 MR diff 与检视意见;候选报告缺失时仅靠源 1
- 源 2 提炼的规则若与源 1 既有规则冲突,以源 1(已落地代码)为准,源 2 条目记入差异摘要待人工裁决
- **置信度**:源码中有明确注释/封装层/版本声明的为高,依据调用模式推断的为中低,源 2 独有条目默认中
- **`约束/` 下 6 个基础文件始终生成**:即使内容为"无",也保留文件声明"本仓遵循全局规范无例外",作为仓级规则骨架;`模式/` 按实际内容生成,允许为空目录

### 输出要求

- 输出路径:`repos/{repo}/.agent/rules/` 目录(始终生成 `_index.md` + `约束/` 子目录;`模式/` 按实际内容生成)
- 严格按模板输出:
  - `templates/rules_index_template.md` → `_index.md`
  - `templates/rules_constraint_template.md` → `约束/*.md`
  - `templates/rules_pattern_template.md` → `模式/*.md`
- **`_index.md` frontmatter** 含 `repo_id`、`last_modified`、`last_modified_by`(写入 `rev-repo-to-rules`)、`file_count`、`confidence`
- **规则文件 frontmatter** 含 `rule_id`(中文或拼音 snake-case 标识,与文件名对应)、`category`(`约束` / `模式`)、`title`、`confidence`、`last_modified`、`last_modified_by`、`sources`(数组,值域 `code-scan` / `mr-candidate`)
- `last_modified` 格式为 ISO 8601 带时区(如 `2026-06-21T14:30:00+08:00`),用于增量模式下 `git log --before="<last_modified>"` 精确定位锚点提交;仅日期精度会导致同日内多次提交无法区分
- 每条规则给出具体的代码证据(`文件路径::符号名` 格式)或代码示例,不使用行号
- `约束/` 下 6 个基础文件始终生成:`第三方库.md` / `接口使用.md` / `编码风格.md` / `性能.md` / `可观测性.md` / `文档同步.md`;无内容的节写"本仓遵循全局规范无例外"
- `模式/` 文件按实际模式生成,命名 `动作.md` 或 `模式名.md`(中文,允许夹必要英文术语);每个文件含:适用场景 / 模式要点 / 涉及修改的文件 / 代码示例 / 常见坑
- 同步在差异摘要中记录:提取条目数、文件拆分情况(新增/删除/刷新的文件清单)、置信度评估、与全局编码规范的差异点
