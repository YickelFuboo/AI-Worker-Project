# 系统级架构设计逆向生成

## 功能描述

**结合历史架构设计文档 + 代码仓逆向的架构元素信息，综合形成系统级架构设计结果**，产出 `architectures/system_architectures.md`。

历史方案是面向人的真正系统设计稿，含切分理由、决策背景、流程编排意图、架构假设、演进路线等**只能由人写出的信息**；代码逆向只能反推"现在是什么样"，反推不出"为什么这么样"。因此本 skill 把两类输入做**双信息域抽取**：

| 信息域 | 主抽取源 | 写入位置 | 缺失时行为 |
|--------|----------|----------|-----------|
| **事实域**：接口签名、依赖关系、协议层、部署形态、容量现状 | 代码 + `.agent/*.md` + 元素层产物 | §3 拓扑 / §4 接口面 / §6 DFX 现状值 / §8 外部域 | 必须有，不应在元素层缺失时执行本 skill |
| **意图域**：切分理由、决策背景、流程编排意图、架构假设、风险预判、演进路线 | `knowledge/历史方案/架构方案/` 下文档 | §1 系统定位 / §2 切分理由 / §5 流程编排意图 / §7 ADR 背景 / §9 风险与演进 | 标注"无历史方案输入，本节仅基于现行代码归纳，置信度降级"并继续产出 |

**分流规则**（事实 vs 意图冲突时）：事实域以代码为准（即使历史方案有不同描述）；意图域以历史方案为准（除非 status=已废弃）；两域不互相覆盖，并存表达。冲突的"事实域分歧"记入差异摘要请人工裁决，不阻断产出。

## 所属 Agent

架构逆向 Agent

## 适用场景

- 完成 `rev-arch-element-extract` 抽出元素后，需把元素层与历史设计文档融合成系统级总览
- 历史架构设计文档新增 / 更新时（如新版 SE 评审纪要、新设计稿归档），系统级文档需同步抽取新意图信息
- 跨 NF 影响分析、对外架构汇报、新成员入职上手时需要一份系统级总图
- 系统级关键架构决策需要沉淀为 ADR 时（产出 `architectures/decisions/ADR-{NNN}-{topic}.md`）

## 工作方式

### 执行步骤

**模式判定**：检查 `architectures/system_architectures.md` 是否存在且 YAML 头部有 `last_modified`。存在 → 增量更新模式；不存在 → 首次生成模式。

#### 首次生成模式

1. **加载事实源（代码逆向元素层）**：
   - 读取 `architectures/logic_view/elements_tree.yaml`，获取所有元素清单与 `depends_on` / `depended_by` 关系
   - 遍历 `architectures/logic_view/elements/{name}/spec.md`：聚合元素定位（§1）、职责（§2）、质量属性（§4）、部署形态（§8）
   - 遍历 `architectures/logic_view/elements/{name}/interfaces.yaml`：归纳系统级对外接口面（SBI / 协议级 / OAM / 暴露面）
   - 遍历 `architectures/logic_view/elements/{name}/dependencies.yaml`：归纳系统级外部依赖（基础设施、内核模块、外部域）
   - **事实域输出位置**：§3 拓扑、§4 接口面、§6 DFX 现状值、§8 外部域

2. **加载意图源（历史架构设计文档，主抽取源）**：
   - 扫描 `knowledge/历史方案/架构方案/` 下所有 `.md` / `.pdf` / `.txt` 文件（doc/docx/pptx 需投递时先用 pandoc 转 md；本 skill 不做格式转换）
   - 按文档 frontmatter（如有）的 `status` 字段过滤：跳过"已废弃"；"已演进"沿 `superseded_by` 链找到现行版本
   - 按内容相关性挑选系统级方案（与 ≥3 个元素相关、或覆盖 ≥1 个端到端流程、或描述系统级 DFX/部署/演进策略）
   - **逐文档抽取四类意图信息**，构建意图域知识表：
     a. **切分理由**：为什么按这些元素切分？哪些是 3GPP 规范继承、哪些是本产品的工程决策？→ 写入 §2
     b. **决策背景**：每个关键架构决策（SBI / CUPS / 用户面下沉 / 数据存储分层 / 非 3GPP 接入分支 等）的当年权衡、候选方案、否决原因 → 写入 §7 各 ADR 的「背景与问题」「候选方案」「决策与理由」
     c. **流程编排意图**：每个端到端流程（UE 注册 / PDU 会话建立 / 切换 / 鉴权 等）当初设计时的关键假设、关键编排理由、关键异常处理意图 → 写入 §5 流程摘要末尾的「编排意图」子条
     d. **架构假设、风险预判、演进路线**：原设计书中明示或暗示的系统级风险（NRF 单点、PFCP 明文、Mongo 共享等）、未来演进方向（多 region / 切片增强 / 边缘下沉 等） → 写入 §9
   - **意图域输出位置**：§1 系统定位、§2 切分理由、§5 流程编排意图、§7 ADR 背景、§9 风险与演进
   - **抽取方式**：每条意图条目记录"来源 solution_name + 章节定位"，便于读者回溯原文；不抄整段，做架构语言的概括转写

3. **双信息域分流与冲突处理**：
   - **事实域**（接口签名、依赖关系、协议层、部署现状、容量现状）：以代码事实源为准
   - **意图域**（理由、背景、假设、演进）：以历史方案为准（除非 status=已废弃）
   - **两域并存表达**：同一章节内可同时含事实陈述与意图陈述，明确标注「现状」与「原设计意图」
   - **冲突识别**：
     a. 事实层冲突（历史方案描述 A，代码事实是 B）→ 事实写代码版本，差异摘要列「历史方案 X 第 Y 章描述为 A，现行实现为 B，建议人工裁决是否更新历史方案或视为合理演进」
     b. 意图层无冲突（历史方案是唯一权威），多方案描述不一致时取最新 status=现行 的方案
   - **历史方案缺失或不足**：意图域章节标注「无历史方案输入，本节仅基于现行代码归纳，置信度降级；建议补齐历史架构设计文档后重跑」，仍产出文档；frontmatter confidence 降为 medium 或 low

4. **系统级章节归纳**（事实 + 意图融合写作）：
   - **§1 系统定位**：事实部分用一段架构语言描述本系统在产业中的位置、核心能力、不做什么；意图部分从历史方案引入「产品愿景 / 战略定位 / 不做什么的边界声明」原文转写
   - **§2 元素分解与切分理由**：表格 14 元素 + element_type + 一句话职责（来自元素 spec §2）+ **切分理由（强抽取自历史方案，无则标注事实推断）** + 所属代码仓
   - **§3 系统级拓扑**：PlantUML 图，含所有元素、所有依赖边、协议层标注（SBI / NGAP / PFCP / GTP-U / IKEv2 等）；可分多个子图（控制面 / 用户面 / 接入面）。事实纯度最高的章节，不夹意图
   - **§4 系统级接口面**：归纳对外暴露的接口面（OAM 北向 / AF-NEF 暴露面 / 计费域接入 / 用户面 N3-N6）；不重复元素级接口详情，给元素跳转。事实纯度最高的章节
   - **§5 端到端流程索引**：列出典型跨 NF 流程，每个流程含：参与元素 + 主要接口（事实）+ 简要时序（事实）+ **编排意图与关键假设（强抽取自历史方案）**
   - **§6 系统级 DFX 策略**：跨元素 DFX（全网 mTLS / OAuth2 / 全局限流 / 灾备 / 重试退避 / 观测埋点）；现状值从元素 spec §4 聚合（事实），目标值与策略原因从历史方案抽取（意图）；两者表格分列
   - **§7 关键架构决策（ADR 索引）**：表格列 ADR 编号 + 主题 + 一句话决策；详情归 `architectures/decisions/ADR-{NNN}-{topic}.md`，**ADR 正文背景与候选方案部分强抽取自历史方案**
   - **§8 外部域整合关系**：与外部实体的系统级关系（MongoDB / Prometheus / OpenTelemetry / gtp5g / xfrm / AF / 计费域 / UE-RAN-WiFi-TWAN）；事实纯度最高的章节
   - **§9 系统级风险与演进**：从历史方案抽取的原设计风险与演进路线（意图） + 元素层暴露的现行风险（事实），两者明示分列

5. **生成 ADR 占位与正文**：识别 §7 中提到的系统级决策，为每个决策生成 `architectures/decisions/ADR-{NNN}-{topic}.md`：
   - 若有历史方案覆盖该决策：抽取背景、候选方案、决策理由、影响代价四节，confidence 写 high 或 medium
   - 若无历史方案覆盖：仅生成 frontmatter + 决策声明占位，正文标注「待人工或历史方案补全」

6. **生成产物**：按模板 `templates/system_architectures_template.md` 输出 `architectures/system_architectures.md`；frontmatter 含 `last_modified` / `last_modified_by` / `element_count` / `adr_count` / `intent_source_count`（采纳的历史方案数）/ `confidence`

#### 增量更新模式

1. **解析时间锚**：读取 `system_architectures.md` 头部 `last_modified`
2. **变更探测（事实源 + 意图源 双路）**：
   - **事实源路**：
     - `architectures/logic_view/elements_tree.yaml` 头部 `last_modified` 晚于本文件 → 元素清单或依赖边变化 → 刷新 §2 现状列、§3 拓扑
     - 任一 `elements/{name}/spec.md` 的 `last_modified` 晚于本文件 → 刷新 §2 现状列、§4、§5 时序、§6 DFX 现状、§8
   - **意图源路**：
     - `knowledge/历史方案/架构方案/` 下任一文档时间晚于本文件，或新增历史方案 → 重新抽取该方案的四类意图信息 → 刷新 §1、§2 切分理由列、§5 编排意图、§6 策略原因列、§7 ADR 背景、§9 演进路线
     - 历史方案 status 变更（如某方案标 已演进 / 已废弃）→ 重新挑选当前现行方案集合，差量刷新意图域章节
3. **章节映射**：
   - 元素清单或 depends_on/depended_by 变化 → §2 现状列 + §3 系统拓扑（事实域）
   - 元素 interfaces.yaml 变化 → §4 系统级接口面（事实域，仅当对外暴露面变化时刷新）
   - 元素 §4 质量属性聚合变化 → §6 现状值列（事实域）
   - 历史方案新增/更新 → §1、§2 切分理由列、§5 编排意图、§6 策略原因列、§7 ADR 背景、§9（意图域）
4. **刷新决策**：无变更仅更新时间戳；局部变更只刷新对应章节与对应信息域；元素数变化 ≥30%、拓扑结构性重排、或核心历史方案首次纳入 → 触发首次生成式全量刷新
5. **更新时间戳与报告**：差异摘要列出本次新增的意图条目、新发现的事实/意图冲突、置信度调整

### 注意事项

- **双信息域并存表达**：本 skill 的核心价值是融合事实与意图；事实陈述与意图陈述在同一章节内并列时必须明示「现状」「原设计意图」标签，避免混淆
- **意图抽取是主动任务，不是装饰**：§2 切分理由、§5 编排意图、§7 ADR 背景、§9 演进路线四处若历史方案有材料而本 skill 未抽取，视为产物不达标
- **历史方案与代码冲突的分流规则**：事实层冲突以代码为准并写差异摘要；意图层取最新 status=现行 的方案；不要让"代码与历史不一致"成为忽略历史方案的借口（演进留痕反而比代码事实更有价值）
- **意图条目必须留来源**：每条抽取自历史方案的意图条目记录"来源 solution_name + 章节定位"（如「参考自 ARCH-2022-03《核心网架构 v3.2》§4.1」），便于读者回溯原文
- **不抢元素层职责**：元素的角色、能力、接口、依赖等已由 `rev-arch-element-extract` 产出；本 skill 只做跨元素聚合 + 历史意图融合，不重复抄写元素级表格行
- **PlantUML 拓扑图必须准确**：协议层标注必须与元素 dependencies.yaml 的 protocol 字段一致；不可虚构边或协议
- **§5 端到端流程索引仅给摘要**：详细序列图本 skill 不强制产出；若需补可另起 `rev-scenario-extract` skill 或人工补 `architectures/scenario_view/`
- **§7 ADR 不只索引**：有历史方案覆盖的决策应在 ADR 正文中抽取背景、候选方案、决策理由；无覆盖的才留占位
- **doc/docx/pptx 历史方案**：投递时先用 pandoc 转 md，本 skill 不内嵌格式转换
- **置信度评估**：事实源完整 + 历史方案充足覆盖（≥80% 章节有意图来源）→ high；事实源完整 + 历史方案部分覆盖 → medium；事实源完整 + 历史方案缺失 → low（章节标注降级，不阻断产出）；事实源不完整 → 不应执行，先补 `rev-arch-element-extract`
- **本 skill 不修改元素层**：发现元素层错误时记入差异摘要请人工或回溯触发 `rev-arch-element-extract` 修正
- **本 skill 不修改历史方案**：历史方案是只读输入，发现描述过时时记入差异摘要请人工裁决

### 输出要求

- 主产物：`architectures/system_architectures.md`，严格按模板 `templates/system_architectures_template.md`
- 附产物：`architectures/decisions/ADR-{NNN}-{topic}.md`：有历史方案覆盖的决策抽取完整正文；无覆盖的仅 frontmatter + 决策声明占位
- frontmatter 含：`doc_id`（固定 `system_architectures`）、`product_name`、`last_modified`、`last_modified_by`（写入 `rev-arch-system-design`）、`element_count`、`adr_count`、`intent_source_count`（本次采纳的历史方案数）、`confidence`
- `last_modified` 格式 ISO 8601 带时区（如 `2026-06-22T14:00:00+08:00`）
- 正文章节顺序固定：§1 系统定位 / §2 元素分解与切分理由 / §3 系统级拓扑（PlantUML）/ §4 系统级接口面 / §5 端到端流程索引 / §6 系统级 DFX 策略 / §7 关键架构决策索引 / §8 外部域整合关系 / §9 系统级风险与演进 / 参考源 / 差异摘要
- **参考源章节**：列出本次执行采纳的所有历史方案 solution_name + 状态 + 主要采纳章节，便于审计
- **差异摘要**（一次性产物，不持久化到 md 正文）：含事实/意图冲突项、置信度降级章节与原因、新增意图条目数、新建 ADR 数
- 章节末"参考源"小节：列出本次执行采纳的历史方案 solution_name 与采纳点
- 输出系统级架构生成报告：识别的元素数、依赖边数、ADR 数、参考源采纳点数、置信度、差异摘要（含与历史方案的冲突项）
