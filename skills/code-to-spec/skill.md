# 代码反向生成 Spec

## 功能描述

对存量代码仓（无 Spec 文档的项目），基于已有代码反向生成功能规格说明书（Spec）。通过分析代码结构、接口定义、数据模型，还原出功能规格、接口规格和数据规格。

## 适用场景

- 存量产品接入 AI 辅助开发，需要补齐 Spec 文档
- 代码与文档不一致，需要从代码事实出发重建 Spec
- 反向流程的第一步：代码 → 分析 → **Spec** → 设计
- 对应设计文档中的"存量产品反向生成 spec"

## 工作方式

### 执行步骤

1. **构建代码知识图谱**（如尚未构建）：调用 `build-graph` 建立代码仓数字孪生
2. **收集项目信息**：
   - 阅读 README.md、Claude.md（如有）
   - 阅读现有文档（如 `architectures/`、`knowledge/` 下的内容）
3. **分析代码架构**：
   - 调用 `read-project-struct` 获取模块树与摘要
   - 识别各模块的职责分工
4. **提取功能清单**：
   - 从模块摘要聚合出业务功能列表
   - 从 API 路由/接口定义提取功能接口清单
5. **提取技术栈**：
   - 从构建文件（package.json、go.mod、pom.xml 等）提取语言、框架、依赖
6. **提取数据模型**：
   - 从代码中的数据定义（struct、class、schema）提取核心数据实体
7. **编写 Spec 文档**：
   - 按 Spec 模板组织输出
   - 标注"由代码反推，待人工确认"的项
8. **输出 Spec 文档**：写入 `requirements/specs/` 目录

### 注意事项

- 从代码反推的 Spec 是"事实描述"而非"设计意图"，可能遗漏业务背景
- 无法从代码中反推的部分（业务目标、历史决策）标注"待补充"
- 接口规格从代码注解/注释中提取，如无注解则标注"缺少文档"
- 如果项目已有部分 Spec 文档，优先更新而非覆盖
- 反推完成后建议人工 Review 确认

### 输出要求

- 仓级 Spec：`requirements/specs/repo_spec.md`
- 模块级 Spec：`requirements/specs/modules/{module_name}_spec.md`
- 所有由代码反推的条目标注 `[INFERRED]` 标记
- 待确认项列表
