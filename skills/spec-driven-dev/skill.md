# 规格驱动开发

## 功能描述

执行严格的三阶段规格驱动开发流程：需求 → 接口规格 → 测试用例 → 代码实现。每一步都有明确的输入、输出和检查点，确保实现与规格一致。

## 适用场景

- 已有完整的功能规格说明书（Spec）和设计文档
- 需要保证代码实现严格符合规格定义
- 需要先写测试再写代码（TDD 风格）
- 正向流程的最终实现阶段

## 工作方式

### 执行步骤

#### 阶段一：需求 → 接口规格

1. 读取功能 Spec 文档中的接口规格定义
2. 提取每个接口的 Request/Response 字段、类型、校验规则
3. 生成接口规格清单（每个接口一个条目）
4. 确认接口规格与设计文档中的模块间接口表一致

#### 阶段二：接口规格 → 测试用例

1. 基于接口规格，为每个接口生成测试用例
2. 覆盖：正常场景、边界值、异常输入、错误码
3. 测试用例采用 Given-When-Then 格式
4. 输出测试用例文件到 `tests/` 目录

#### 阶段三：Spec + 测试 → 代码

1. 读取设计文档中的模块划分和类/函数表
2. 基于接口规格和测试用例，实现代码
3. 实现完成后运行测试，验证通过
4. 如有测试失败，修复代码直到全部通过

### 注意事项

- 严格按三个阶段顺序执行，不可跳跃
- 每个阶段完成后，确认输出物完整再进入下一阶段
- 测试用例是代码实现的"护栏"，必须先于代码完成
- 如果实现过程中发现设计问题，回退到设计阶段修改，不可就地打补丁
- 实现完成后必须运行测试，报告结果

### 输出要求

- 接口规格清单：`requirements/specs/{feature_name}_api_spec.md`
- 测试用例：`tests/{feature_name}_test.md`（或对应测试代码文件）
- 代码实现：按设计文档中定义的模块路径
- 测试结果报告
