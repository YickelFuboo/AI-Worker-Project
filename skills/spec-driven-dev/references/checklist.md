# 规格驱动开发参考

## 三阶段流程检查清单

### 阶段一检查项
- [ ] 每个接口有明确的 Method、Path、Request/Response 定义
- [ ] 每个字段标注类型、必填/可选、校验规则
- [ ] 每个接口列出所有可能的错误码
- [ ] 接口规格与设计文档的模块间接口表一致

### 阶段二检查项
- [ ] 每个接口至少覆盖正常场景、一个边界场景、一个异常场景
- [ ] 测试用例使用 Given-When-Then 格式
- [ ] 测试数据覆盖字段校验规则的所有边界
- [ ] 错误码场景都有对应的测试用例

### 阶段三检查项
- [ ] 所有测试用例通过
- [ ] 代码实现符合设计文档的模块划分
- [ ] 接口签名与 Spec 定义一致
- [ ] 无遗留的 TODO 或占位符

## 停止原则

以下情况应暂停实现，回到上游阶段修正：
- 实现过程中发现接口规格不足以覆盖用户场景 → 回退到阶段一
- 测试用例无法覆盖某个规格项（规格不可测试） → 回退到阶段一修正规格
- 设计文档的模块划分导致实现时耦合过重 → 回退到设计阶段
- 发现需要新增外部依赖但未在设计文档中说明 → 回退到设计阶段

## 修复原则

- 优先修复规格问题，而非绕过规格实现
- 测试失败时，先确认是测试写错了还是实现错了
- 不可为通过测试而弱化测试断言
- 修复后必须运行全部测试，不只看失败的测试
