# 回归测试选择

## 功能描述

基于需求继承性分析和架构影响域分析的结果，从全量测试用例中选择本次需求迭代需要执行的回归测试范围。避免盲目全量回归，同时确保不遗漏受影响的存量功能测试。

## 所属 Agent

测试设计 Agent

## 适用场景

- 需求变更涉及修改已有特性或接口
- 需要确定回归测试范围
- CI/CD 流水线需要按变更范围选择性执行测试
- 版本发布前的回归测试规划

## 工作方式

### 执行步骤

1. **读取继承性报告**：加载 `inheritance-analysis` 输出的兼容性风险和回归场景清单
2. **读取架构影响域**：加载 `arch-impact-analysis` 输出的受影响架构元素和接口
3. **读取跨视角缓存**：加载 `.agent/cross_perspective_cache.yaml`，获取场景→接口→代码的映射
4. **读取测试覆盖映射**：加载 `tests/coverage_mapping.yaml`，获取测试用例→场景→架构元素的映射
5. **选择回归用例**：
   - 直接回归：继承性报告列出的回归场景对应的测试用例
   - 间接回归：受影响接口的所有调用方场景对应的测试用例
   - 风险加权：按风险等级（高/中/低）排序，高优先级必选
6. **生成回归测试清单**：输出按优先级分组的回归用例 ID 列表

### 注意事项

- 回归范围选择必须基于客观的影响分析，不可主观裁剪
- 高风险项（接口契约变更、共享状态修改）的回归用例不可省略
- 若继承性分析标记为高风险但无对应测试用例，应标注为"测试盲区"并触发补充
- 跨仓影响需通过 `cross-repo-impact` 的输出扩展回归范围

### 输出要求

- 输出回归测试清单：按优先级（高/中/低）分组的测试用例 ID
- 标注每个用例的回归原因（直接受影响/间接影响/风险加权）
- 输出测试盲区清单（受影响但无对应测试的场景）
