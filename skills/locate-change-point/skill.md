# 改码点定位

## 功能描述

执行完整的分层读码策略（L1 → L2 → L3），从用户的自然语言需求出发，通过架构视图、文件框架视图、源码视图的递进搜索，最终定位到明确的改码点（文件路径 + 类名/函数名）。

## 适用场景

- 用户提出自然语言需求后，需要找到应修改的具体代码位置
- 正向流程中从需求到代码的关键衔接环节
- 需要确认"改哪里"之后再开始"怎么改"
- 对应设计文档 REQ-CG-D05：需求 → 改码点定位流程

## 工作方式

### 执行步骤

#### L1：架构定位

1. 调用 `read-project-struct`（本目录 skill）获取项目模块树与摘要
2. 将用户需求中的关键词与模块摘要做语义匹配
3. 锁定 1-3 个最相关的候选模块/目录

#### L2：符号定位

1. 对候选模块中的关键文件，调用 `read-file-summary`（本目录 skill）获取文件框架
2. 将需求关键词与类/函数摘要做语义匹配
3. 锁定 1-3 个最相关的候选类/函数

#### L3：源码确认

1. 调用 `read-function-code`（本目录 skill）获取候选符号的完整源码
2. 确认该符号确实是需要修改的位置
3. 确认修改范围（改函数体、改签名、改类结构等）

#### 可选：依赖分析

1. 如需评估修改影响面，查询调用方/被调方
2. 标记需要连带修改的上下游符号

#### 输出改码点

输出格式：`(file_path, class_name | function_name, 修改类型, 修改说明)`

### 注意事项

- **严格遵循 L1 → L2 → L3 顺序**，禁止跳过中间层直接读取源码
- 每层只调用一次对应 Tool，根据返回结果再决策下一步
- 如果 L1 无法收敛到具体模块，向用户确认需求描述
- 如果 L2 无法锁定符号，扩大候选范围或请求用户提示
- 输出改码点时附带简要定位理由（为什么是这个符号）

### 输出要求

```
改码点定位结果:

1. 文件: {file_path}
   符号: {class_name}.{method_name} 或 {function_name}
   修改类型: 修改函数体 / 修改签名 / 新增方法 / ...
   定位理由: {从 L1/L2 摘要匹配到该符号的推理链}
   影响面: {如需连带修改的上下游列表}
```
