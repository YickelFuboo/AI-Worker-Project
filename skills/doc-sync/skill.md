# 文档同步更新

## 功能描述

当代码变更涉及接口行为、项目结构、数据模型等关键要素时，自动检查并同步更新相关文档。确保代码与文档始终保持一致，避免文档腐烂。

## 适用场景

- 修改了 API 接口的请求/响应结构
- 新增/删除了模块或文件
- 修改了数据结构定义
- 修改了构建配置或外部依赖
- 代码重构后需要同步更新设计文档
- 对应参考规则中的 `doc-sync`

## 工作方式

### 执行步骤

1. **识别变更类型**：分析本次修改涉及哪些类型的变更
   - 接口变更：API 函数签名、Request/Response 结构
   - 结构变更：目录/模块/文件增删
   - 数据变更：struct/class 字段修改
   - 依赖变更：新增/升级/移除依赖
   - 配置变更：构建脚本、环境变量

2. **确定需同步的文档**：根据变更类型确定影响范围

   | 变更类型 | 需同步的文档 |
   |----------|-------------|
   | 接口变更 | Spec（接口规格）、Design（模块间接口表）、README（API 说明） |
   | 结构变更 | Design（模块划分）、Spec（目录概览）、Claude.md（代码定位指南） |
   | 数据变更 | Spec（数据规格）、Design（数据结构表） |
   | 依赖变更 | Spec（技术栈、外部依赖） |
   | 配置变更 | README、Claude.md |

3. **执行同步更新**：
   - 读取需同步的文档当前版本
   - 根据代码变更更新对应章节
   - 保持文档模板格式不变

4. **记录同步日志**：
   - 在变更日志中记录本次文档同步的详情

### 注意事项

- 只同步与本次变更相关的文档章节，不做全文档重写
- 如果文档中有"由代码反推"的标记，确认后移除或更新
- 如果文档不存在（存量项目首次接入），建议执行 `code-to-spec` 或 `code-to-design` 生成
- 同步后应标注更新时间和更新原因
- 不要创建不必要的文档

### 输出要求

- 更新受影响的文档文件
- 变更日志中记录文档同步项：`{date} - 文档同步: {变更描述} → 更新了 {文档列表}`
