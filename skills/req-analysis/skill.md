# 需求分析

## 功能描述

将用户的自然语言业务诉求转化为结构化需求文档。分析需求背景、目标、范围、场景与约束，为后续编写规格说明书和设计文档提供输入。

## 适用场景

- 用户提出模糊或宽泛的业务需求（如"给缓存加TTL"、"支持多租户"）
- 需要将一段自然语言描述拆解为可执行的功能需求项
- 需要明确需求边界、适用角色、关联系统
- 正向流程的第一步：需求 → 规格 → 设计 → 代码

## 工作方式

### 执行步骤

1. **收集上下文**：阅读项目 README.md、Claude.md（如有），了解项目定位与技术栈
2. **理解诉求**：与用户确认需求背景、目标用户、期望结果
3. **场景拆解**：识别用户场景（正常流程、异常流程、边界条件）
4. **需求结构化**：按模板输出需求分析文档，包含功能点、非功能要求、验收标准
5. **边界确认**：明确在范围外的事项，避免后续范围蔓延

### 注意事项

- 遇到需求中的关键不确定性（边界、兼容性、输入输出格式），必须先提问再下笔
- 需求分析聚焦"做什么"而非"怎么做"，不涉及技术实现细节
- 每个需求项应有明确的验收标准（Given-When-Then 格式）
- 文档输出到 `requirements/` 目录下

### 输出要求

- 文档路径：`requirements/{feature_name}_requirements.md`
- 包含完整的功能概述、用户场景、用例列表、业务规则、验收标准
- 如有不明确项，在文档中标注"待确认"
