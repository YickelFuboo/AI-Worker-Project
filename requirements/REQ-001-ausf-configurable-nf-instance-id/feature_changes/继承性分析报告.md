# 场景继承性分析报告：AUSF 支持配置 nfInstanceId

## 1. 分析概述

| 项目 | 内容 |
|------|------|
| 关联需求 | REQ-001-ausf-configurable-nf-instance-id |
| 业务变更来源 | `requirements/REQ-001-ausf-configurable-nf-instance-id/feature_changes/**` |
| 场景数量 | 3 |
| 存量影响结论 | 兼容扩展，并存在配置错误处理路径的中风险待确认项 |
| 最高兼容性风险 | medium |

本次 03 分析基于 02 生成的 3 个业务变更说明文件。需求新增 AUSF 可选 `nfInstanceId` 配置能力，明确未配置时保持原有自动生成行为，并新增非法配置启动失败规则。整体判断为对存量未配置启动场景的兼容扩展；主要兼容性风险集中在“是否存在历史部署或验收口径默认允许非法/未知配置继续启动”的待确认项。

## 2. 输入场景清单

| 场景ID | 业务变更ID | 变更类型 | 一级特性 | 子特性 | 变更摘要 |
|--------|------------|----------|----------|--------|----------|
| SCENARIO_001 | FC-001 | new | 鉴权与安全特性 | 基于AUSF_UDM的SNPN鉴权 | AUSF 支持在配置文件中声明合法 UUID v4 的 `nfInstanceId` 并使用该配置值作为稳定 NF Instance ID。 |
| SCENARIO_002 | FC-002 | modify | 鉴权与安全特性 | 基于AUSF_UDM的SNPN鉴权 | AUSF 在未配置 `nfInstanceId` 时继续保持原有自动生成 NF Instance ID 的行为。 |
| SCENARIO_003 | FC-003 | new | 鉴权与安全特性 | 基于AUSF_UDM的SNPN鉴权 | AUSF 在配置的 `nfInstanceId` 非法或不是 UUID v4 时启动失败并提示配置错误。 |

## 3. 存量场景影响分析

### 3.1 SCENARIO_001 运维人员为 AUSF 配置稳定 nfInstanceId 后启动

| 项 | 内容 |
|----|------|
| 业务变更ID | FC-001 |
| 影响类型 | compatible_extension |
| 风险等级 | low |
| 受影响存量场景 | AUSF 启动时生成 NF Instance ID 的存量场景；外部检查、监控或运维追踪观察 AUSF NF Instance ID 的存量场景 |
| 受影响业务规则 | 未配置 `nfInstanceId` 时自动生成 NF Instance ID；外部可观察 NF Instance ID 用于部署和运维识别 |
| 兼容性结论 | 新增合法配置路径是对存量 AUSF 启动身份能力的兼容扩展；在未配置场景保持原行为的前提下，不破坏存量部署。 |

#### 影响说明

SCENARIO_001 新增了“配置值优先”的业务路径：当运维人员在 AUSF 配置文件中提供合法 UUID v4 的 `nfInstanceId` 时，AUSF 使用该配置值作为自身 NF Instance ID。该路径改变的是新增配置场景下的可观察身份来源，不改变未配置存量场景的触发条件和默认行为。因此它对存量场景的影响类型为兼容扩展。

该场景可能影响运维侧对 AUSF NF Instance ID 的观察和追踪习惯：配置后，重启前后的 ID 预期保持稳定；未配置时仍不保证稳定。该差异需要在后续验收和说明中区分清楚。

#### 继承性判断依据

- 来源业务变更：`requirements/REQ-001-ausf-configurable-nf-instance-id/feature_changes/基于AUSF_UDM的SNPN鉴权/SR-001-配置稳定nfInstanceId_变更说明.md`
- 存量特性资料：`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/feature.yaml`、`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/spec.md`
- 判断依据：现有子特性资料关联 AUSF 架构元素，但未描述可配置 NF Instance ID 场景；02 明确该能力为新增配置路径，且需求要求未配置时保持原有自动生成行为。

### 3.2 SCENARIO_002 未配置 nfInstanceId 时 AUSF 保持自动生成行为

| 项 | 内容 |
|----|------|
| 业务变更ID | FC-002 |
| 影响类型 | compatible_extension |
| 风险等级 | low |
| 受影响存量场景 | AUSF 未配置 `nfInstanceId` 时自动生成 NF Instance ID 的存量启动场景；现有未修改配置文件的部署升级场景 |
| 受影响业务规则 | 未配置 `nfInstanceId` 时自动生成 NF Instance ID；现有 AUSF 配置文件继续可用 |
| 兼容性结论 | 该场景显式保护存量行为，是本需求的主要兼容性约束；只要未配置路径保持自动生成，即不破坏现有部署。 |

#### 影响说明

SCENARIO_002 对存量场景不是破坏性修改，而是把原有默认行为纳入新增配置能力后的兼容性规则。它要求现有部署方不增加 `nfInstanceId` 时，AUSF 仍按原有行为自动生成 NF Instance ID，并且升级到支持该配置项的版本后不需要额外配置即可继续启动。

该场景应作为后续阶段的核心回归范围。若后续设计或实现把 `nfInstanceId` 配置项误处理为必填项，或在未配置时改变启动结果，就会违反该兼容性承诺。

#### 继承性判断依据

- 来源业务变更：`requirements/REQ-001-ausf-configurable-nf-instance-id/feature_changes/基于AUSF_UDM的SNPN鉴权/SR-002-未配置nfInstanceId保持自动生成_变更说明.md`
- 存量特性资料：`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/feature.yaml`、`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/spec.md`
- 判断依据：需求文档 BR-003 明确未配置 `nfInstanceId` 时必须保持原有每次启动自动生成 UUID 的行为；02 将该场景记录为 modify，是对存量默认行为的兼容性约束。

### 3.3 SCENARIO_003 配置的 nfInstanceId 不是合法 UUID v4

| 项 | 内容 |
|----|------|
| 业务变更ID | FC-003 |
| 影响类型 | potential_conflict |
| 风险等级 | medium |
| 受影响存量场景 | AUSF 配置错误处理相关存量场景；包含未知或额外配置项的现有配置启动场景 |
| 受影响业务规则 | 非法 `nfInstanceId` 必须启动失败并提示配置错误；现有配置文件继续可用 |
| 兼容性结论 | 对未声明 `nfInstanceId` 的存量部署无影响；对声明了非法或非 UUID v4 值的配置场景会形成新的启动失败规则，需要确认是否存在历史部署或验收口径与该规则冲突。 |

#### 影响说明

SCENARIO_003 新增了非法配置失败路径。该规则有助于避免 AUSF 使用不符合 UUID v4 要求的 NF Instance ID，但也可能改变部分错误配置场景的启动表现。若历史上存在包含未知配置项、错误值或非 UUID v4 标识值但仍可启动的部署假设，则本场景可能与这些隐含存量行为产生冲突。

由于需求要求 `nfInstanceId` 是新增可选配置项，未声明该项的配置文件不受影响；风险只集中在主动声明该配置项但值非法的场景。因此该风险不阻断后续分析，但需要后续阶段继续保留并确认。

#### 继承性判断依据

- 来源业务变更：`requirements/REQ-001-ausf-configurable-nf-instance-id/feature_changes/基于AUSF_UDM的SNPN鉴权/SR-003-非法nfInstanceId启动失败_变更说明.md`
- 存量特性资料：`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/feature.yaml`、`features/鉴权与安全特性/基于AUSF_UDM的SNPN鉴权/spec.md`
- 判断依据：需求文档 BR-004、BR-005 明确合法值必须满足 UUID v4，非法或非 UUID v4 时必须启动失败并提示配置错误；现有特性资料未描述该配置错误路径。
