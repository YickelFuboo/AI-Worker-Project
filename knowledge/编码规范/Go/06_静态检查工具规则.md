# Go 静态检查工具规则

> 来源：
> - gofmt: https://pkg.go.dev/cmd/gofmt
> - go vet: https://pkg.go.dev/cmd/vet
> - golangci-lint: https://golangci-lint.run/
>
> 形式：上方中文简要 + 下方全文（工具能力、规则清单、配置示例）。
> 适用：所有 Go 项目应在 CI 中至少跑 `gofmt`/`go vet`/`golangci-lint`。

---

## 中文简要

Go 静态检查生态有三层：
1. **gofmt/goimports**：格式化工具，所有 Go 项目强制使用，零配置。
2. **go vet**：Go 工具链内置的 suspicious construct 检查器，约 30 个 analyzer，默认全开。
3. **golangci-lint**：聚合 100+ linter 的第三方工具，并行执行、缓存结果，是社区事实标准。

核心要点：

- **gofmt**：自动格式化代码（缩进、空格、对齐）。`gofmt -w .` 写回文件；`gofmt -d .` 只看 diff。`goimports` 是 `gofmt` 超集，额外管理 import。
- **go vet**：默认运行全部 analyzer；用 `-<analyzer>=false` 关闭单个。常用 `go vet ./...`。常见能抓的问题：`printf` 格式串与参数不匹配、`sync.Mutex` 值复制、循环变量捕获、unbuffered `os.Signal` channel、`time.Format` 用 `2006-02-01`、unreachable code、`errors.As` 传非指针等。
- **golangci-lint**：默认启用的 linter 为 `errcheck`/`gosimple`/`govet`/`ineffassign`/`staticcheck`/`unused`；常用额外启用 `gofmt`/`goimports`/`revive`/`gosec`/`gocyclo`/`misspell`/`nakedret`/`prealloc`/`unconvert`/`unparam`/`nolintlint`/`depguard`/`gocritic`/`bodyclose`/`rowserrcheck`/`sqlclosecheck`/`noctx`/`nilerr`/`wastedassign` 等。
- **配置**：在仓库根放 `.golangci.yml`，定义 `linters.enable/disable`、`run.timeout`、`issues.max-issues-per-linter` 等。
- **CI 集成**：建议 `make lint` 跑 `golangci-lint run`，并把 `gofmt -l .` 与 `go vet ./...` 作为独立检查项。

> 推荐顺序：本地用 `goimports`（保存时自动跑）→ 提交前 `go vet ./...` → CI 跑 `golangci-lint run`。

## 全文

### 1. gofmt / goimports

#### 能力
- `gofmt`：自动格式化 Go 源文件——缩进用 tab、操作符对齐、空格规范化。零配置。
- `goimports`：`gofmt` 超集，额外自动增删 import 语句。

#### 常用命令
```bash
# 格式化并写回
gofmt -w .

# 只显示 diff，不写回
gofmt -d .

# 列出需要格式化的文件
gofmt -l .

# goimports（需安装：go install golang.org/x/tools/cmd/goimports@latest）
goimports -w .
goimports -d .
goimports -l .
```

#### 编辑器集成
- VS Code：`"editor.formatOnSave": true` + `"go.formatTool": "goimports"`。
- GoLand：Preferences → Tools → File Watchers → goimports。
- Vim：`vim-go` 插件默认集成。

#### 关键规则
- 缩进统一 tab。
- 大括号不换行（与关键字同行）。
- 结构体字段对齐。
- 运算符两侧空格一致。
- 不改变字符串内容、不重排字段。

### 2. go vet

#### 概述
> Vet examines Go source code and reports suspicious constructs, such as Printf calls whose arguments do not align with the format string. Vet uses heuristics that do not guarantee all reports are genuine problems, but it can find errors not caught by the compilers.

#### 用法
```bash
# 当前目录
go vet

# 指定包
go vet my/project/...

# 列出可用 analyzer
go tool vet help

# 查看某 analyzer 详情
go tool vet help printf
```

退出码：非零表示调用错误或发现问题，0 表示无问题。注意：vet 不检查所有可能问题，依赖启发式，应作为指导而非正确性硬指标。

#### 内置 analyzer 清单（Go 1.22+）

| Analyzer | 用途 |
|---|---|
| `appends` | 检查 `append` 后缺失值 |
| `asmdecl` | 报告汇编文件与 Go 声明的不匹配 |
| `assign` | 检查无用赋值 |
| `atomic` | 检查 `sync/atomic` 常见错误 |
| `bools` | 检查布尔运算符常见错误 |
| `buildtag` | 检查 `//go:build` 与 `// +build` 指令 |
| `cgocall` | 检测 cgo 指针传递规则违规 |
| `composites` | 检查未键值化的复合字面量 |
| `copylocks` | 检查锁被错误地按值传递 |
| `defers` | 报告 defer 常见错误 |
| `directive` | 检查 Go 工具链指令如 `//go:debug` |
| `errorsas` | 报告传非指针/非 error 给 `errors.As` |
| `framepointer` | 报告汇编在保存前破坏帧指针 |
| `hostport` | 检查传给 `net.Dial` 的地址格式 |
| `httpresponse` | 检查 HTTP 响应使用错误 |
| `ifaceassert` | 检测不可能的 interface-to-interface 类型断言 |
| `loopclosure` | 检查嵌套函数对循环变量的引用 |
| `lostcancel` | 检查 `context.WithCancel` 返回的 cancel 是否被调用 |
| `nilfunc` | 检查函数与 nil 的无用比较 |
| `printf` | 检查 Printf 格式串与参数一致性 |
| `shift` | 检查等于或超过整数宽度的移位 |
| `sigchanyzer` | 检查 `os.Signal` 的无缓冲 channel |
| `slog` | 检查无效结构化日志调用 |
| `stdmethods` | 检查知名接口方法签名 |
| `stdversion` | 报告使用过新的标准库符号 |
| `stringintconv` | 检查 `string(int)` 转换 |
| `structtag` | 检查 struct 字段 tag 是否符合 `reflect.StructTag.Get` |
| `testinggoroutine` | 报告测试启动 goroutine 中调 `(*testing.T).Fatal` |
| `tests` | 检查测试与 example 常见误用 |
| `timeformat` | 检查 `time.Format`/`time.Parse` 用 `2006-02-01` |
| `unmarshal` | 报告传非指针/非接口给 unmarshal |
| `unreachable` | 检查不可达代码 |
| `unsafeptr` | 检查 `uintptr` 到 `unsafe.Pointer` 的无效转换 |
| `unusedresult` | 检查某些函数调用结果的未使用 |
| `waitgroup` | 检查 `sync.WaitGroup` 误用 |

#### 默认行为
默认全部 analyzer 都运行。若显式设置任一 flag 为 true，仅运行那些；若显式设置任一为 false，仅禁用那些。例：`-printf=true` 仅运行 printf；`-printf=false` 运行除 printf 外全部。

#### 核心命令行 flag
```
-c=N            显示问题行加 N 行上下文
-json           以 JSON 输出诊断
```

### 3. golangci-lint

#### 概述
> golangci-lint is a fast Go linters runner. It runs linters in parallel, uses caching, supports YAML configuration, and includes over a hundred linters.

#### 安装
```bash
# 二进制
curl -sSfL https://raw.githubusercontent.com/golangci/golangci-lint/master/install.sh | sh -s -- -b $(go env GOPATH)/bin v1.59.1

# Go install
go install github.com/golangci/golangci-lint/cmd/golangci-lint@v1.59.1

# Homebrew (macOS)
brew install golangci-lint
```

#### 常用命令
```bash
# 在当前项目运行所有启用的 linter
golangci-lint run

# 指定路径
golangci-lint run ./...

# 列出启用的 linter
golangci-lint linters

# 显示配置帮助
golangci-lint config --help

# 缓存管理
golangci-lint cache clean
golangci-lint cache status
```

#### 默认启用的 linter
| Linter | 用途 |

#### 推荐 `.golangci.yml` 配置

```yaml
# .golangci.yml — 中型项目推荐配置
run:
  timeout: 5m
  go: "1.22"
  modules-download-mode: readonly

linters:
  disable-all: true
  enable:
    # 默认六件套
    - errcheck
    - gosimple
    - govet
    - ineffassign
    - staticcheck
    - unused
    # 格式化
    - gofmt
    - goimports
    # 代码质量
    - revive
    - gocritic
    - gocyclo
    - unconvert
    - unparam
    - wastedassign
    - nakedret
    - prealloc
    # 安全与正确性
    - gosec
    - bodyclose
    - noctx
    - nilerr
    - errorlint
    - rowserrcheck
    - sqlclosecheck
    # 文档与拼写
    - misspell
    # 指令规范
    - nolintlint

linters-settings:
  gocyclo:
    min-complexity: 15
  gocritic:
    enabled-tags:
      - performance
      - style
    disabled-checks:
      - hugeParam
      - rangeValCopy
  lll:
    line-length: 140
  funlen:
    lines: 80
    statements: 50
  misspell:
    locale: US
  revive:
    severity: warning
    rules:
      - name: blank-imports
      - name: context-as-argument
      - name: context-keys-type
      - name: dot-imports
      - name: error-return
      - name: error-strings
      - name: error-naming
      - name: exported
        arguments: ["disableStutteringCheck"]
      - name: increment-decrement
      - name: range
      - name: receiver-naming
      - name: time-naming
      - name: var-declaration
      - name: var-naming
  staticcheck:
    checks: ["all", "-ST1000", "-ST1003", "-ST1016", "-ST1020", "-ST1021", "-ST1022"]
  gosec:
    excludes:
      - G104  # unhandled error (errcheck 已覆盖)
    config:
      global:
        audit: false

issues:
  exclude-rules:
    - path: _test\.go
      linters:
        - gosec
        - errcheck
        - gocyclo
        - goconst
        - gochecknoinits
        - gochecknoglobals
    - path: cmd/
      linters:
        - gochecknoinits
  max-issues-per-linter: 0
  max-same-issues: 0
  fix: false

output:
  formats:
    - format: colored-line-number
  sort-results: true
  print-issued-lines: true
  print-linter-name: true
```

#### Makefile 集成示例
```makefile
.PHONY: lint vet fmt check

GO := go
GOLANGCI_LINT := golangci-lint

fmt:
	@$(GOLANGCI_LINT) run --fix ./... || true
	@go mod tidy

vet:
	@$(GO) vet ./...

lint:
	@$(GOLANGCI_LINT) run ./...

check: fmt vet lint
	@echo "All checks passed."

# CI 入口
ci: check test
```

#### CI 集成
GitHub Actions 示例：

```yaml
# .github/workflows/lint.yml
name: lint
on:
  push:
    branches: [main]
  pull_request:
jobs:
  golangci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'
      - uses: golangci/golangci-lint-action@v6
        with:
          version: v1.59
          args: --timeout 5m
```

#### 静默 linter
在代码行上加 `//nolint:lintername` 注释（如 `//nolint:errcheck`），但仅在确实有理由时使用，并用 `//nolintlint` 检查其使用是否合理。

### 5. 推荐流水线

1. **本地保存时**：`goimports` 自动格式化（编辑器集成）。
2. **提交前**：`go vet ./...` + `golangci-lint run ./...`（pre-commit hook）。
3. **CI**：`golangci-lint run` + `go test -race -coverprofile=cover.out ./...` + 覆盖率门禁。
4. **定期**：升级 golangci-lint 版本，启用新 linter。

### 6. 与本仓库 free5GC NF 的适配建议

free5GC 各 NF 子模块独立 `go.mod`，建议：
- 每个 NF 子目录放自己的 `.golangci.yml`（或顶层共享配置 + `//go:build` 路径排除）。
- 注意 free5GC 大量使用生成的 openapi 代码，需在 `issues.exclude-rules` 中按路径 `path: openapi/` 排除大部分 linter。
- NAS/NGAP/PFCP 协议消息编解码常含大 struct 与 magic number，可适当放宽 `gocyclo`/`funlen`/`lll` 阈值。
- `errcheck` 在协议处理路径可能产生大量误报，可在测试代码外保留默认，针对 `internal/sbi/consumer/` 等关键路径单独启用。
