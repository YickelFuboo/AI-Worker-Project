# 历史方案归档

存量产品历次架构与系统设计方案的归档目录，承接从外部（研发库 / Confluence / SE 评审纪要 / 历史 docx-pdf 设计书）导入的方案文档，供后续参考。

## 目录划分

| 子目录 | 承载内容 | 边界 |
|--------|---------|------|
| `架构方案/` | 元素图谱、跨 NF 拓扑、跨仓服务化分解、关键架构决策 | 回答"系统由什么元素构成、如何切分" |
| `系统方案/` | 部署形态、容量规格、关键流程编排、DFX 目标、上线/演进路径 | 回答"系统在什么环境跑、跑多大、怎么运作" |

直接把方案文件放在对应子目录下即可，建议命名保留时间与方案名信息（如 `2023Q3-AMF移动性架构.md`）。同一方案多版本演进时，可建子目录 `{方案名}/v1.md`、`v2.md` 串接。

## frontmatter 模板（建议填写，便于检索与关联）

```yaml
---
solution_name: <方案名>
solution_type: 架构方案 | 系统方案
era: <生效时段，如 2021-2023>
source: <来源平台与原始链接/路径>
status: 现行 | 已演进 | 已废弃
related_elements: [<元素名列表，引用 architectures/logic_view/elements_tree.yaml>]
related_repos: [<仓名列表>]
superseded_by: <如已演进，指向后继方案文件名>
last_modified: "YYYY-MM-DDTHH:MM:SS+08:00"
---
```

## 与其他知识目录的边界

- `历史决策/`：单点架构/技术决策（ADR 风格），通常粒度小、单一议题。历史方案是组合体，含多个决策的整体方案书。
- `领域知识/`：协议规范、产业标准、通用领域知识。历史方案是本产品的私有沉淀。
- `编码规范/`：代码风格与实现约束，与方案无关。
- `质量门禁/`：CI 与发布门禁规则，与方案无关。
